# .agent.md — Projeto Hibit

## 1. Objetivo do agente

Este agente deve ajudar a criar e evoluir o site institucional da empresa **Hibit**, com foco em:

- **Backend** em **.NET 8**
- **Frontend** em **Angular**
- **Banco de dados** em **MySQL**
- **Mensageria** com **RabbitMQ**
- **Deploy** na **King.host**
- **CI/CD** com **GitHub Actions**

O agente deve priorizar:

- arquitetura limpa
- segurança da informação
- LGPD
- testes automatizados
- manutenção fácil
- código legível e sustentável

---

## 2. Escopo do site

O site terá inicialmente estas páginas:

- **Home**
  - apresentação da empresa
  - destaques institucionais e comerciais

- **Sobre**
  - história da empresa
  - missão, visão, valores e diferenciais

- **Contato**
  - formulário de contato
  - mapa
  - envio de mensagem para contato@hibit.com.br
  - a mensagem não deve ser salva no banco
  - a mensagem deve ser criptografada antes de ser enviada ao RabbitMQ
  - o processamento posterior será feito por outra aplicação de controle

### 3.2 E-mail

## 6. Banco de dados

- Usar MySQL
- Usar Entity Framework Core
- Usar migrations
- **Aplicar migrations automaticamente no startup** (`DatabaseInitializer`): cria o database MySQL se não existir e aplica migrations pendentes
- Evitar SQL cru sempre que possível
- Não usar concatenação de strings em consultas
- Nunca expor credenciais no código
- O formulário de contato não deve ser salvo no banco
- O AppDbContext deve conter DbSet para as entidades do sistema
- Novas tabelas devem ser criadas de forma organizada e explícita
- Tabela `usuarios`: armazena o **usuário de aplicação** (credenciais para o frontend acessar a API via JWT)
- Senhas na tabela `usuarios` devem ser armazenadas apenas como hash (nunca em texto puro)

## 7. Segurança da informação

### Obrigatório

- Usar HTTPS
- Proteger contra SQL Injection
- Evitar XSS
- Evitar CSRF
- Validar entrada de dados no backend e no frontend
- Não hardcodar segredos
- Usar variáveis de ambiente para tudo que for sensível
- Não logar PII
- Aplicar o princípio do menor privilégio
- Proteger endpoints da API com **JWT Bearer**
- O frontend deve obter token via `POST /api/auth/login` e enviar `Authorization: Bearer {token}` nas requisições
- Armazenar o token no frontend em `sessionStorage` (não em `localStorage`)

### Recomendado

- Configurar Cloudflare para proteção adicional
- Usar headers de segurança HTTP
- Restringir CORS apenas ao frontend autorizado
- Tratar formulários com cuidado para evitar abuso e vazamento de dados

### LGPD

- Exibir aviso de privacidade
- Solicitar consentimento explícito no formulário
- Coletar apenas o necessário
- Não armazenar dados desnecessários
- Garantir tratamento seguro das mensagens

## 8. RabbitMQ

O fluxo do contato deve ser:

1. Usuário envia o formulário
2. Backend valida os dados
3. Backend criptografa o payload com AES-256
4. Backend publica a mensagem na fila RabbitMQ
5. Outra aplicação consome e processa a mensagem

### Regras

- Não persistir os dados do contato em banco
- Criptografar o payload antes do envio
- Ler chaves e segredos por variável de ambiente
- Separar claramente produtor e consumidor

## 9. Variáveis de ambiente

Sempre que houver valores que possam mudar entre ambientes, usar variáveis de ambiente.

### Exemplos

- string de conexão do banco
- host do RabbitMQ
- usuário e senha do RabbitMQ
- chave de criptografia AES-256
- IV de criptografia
- origem permitida no CORS
- URLs públicas
- credenciais de SMTP
- tokens e segredos de deploy
- `Jwt__Secret`, `Jwt__Issuer`, `Jwt__Audience`, `Jwt__ExpirationMinutes`
- `AppUser__Username`, `AppUser__Password` (seed do usuário de aplicação)

### Regras

- Nunca hardcodar segredos **de produção**
- Nunca deixar senha de produção no repositório
- Sempre criar arquivo de exemplo chamado `.env.example`
- Credenciais de **Docker local** (desenvolvimento) ficam documentadas abaixo e em `.env.example`

## 18. Autenticação JWT — usuário de aplicação

O site institucional usa um **usuário de aplicação** (não é login de visitante). O frontend Angular autentica na API e armazena o JWT para chamar endpoints protegidos.

### Fluxo

1. Ao iniciar, o frontend chama `POST /api/auth/login` com usuário/senha de aplicação
2. A API valida credenciais na tabela `usuarios` e retorna `{ accessToken, expiresAt }`
3. O frontend guarda o token em `sessionStorage`
4. Um interceptor HTTP envia `Authorization: Bearer {token}` em todas as requisições `/api/*` (exceto login)
5. Endpoints protegidos (ex.: `POST /api/contact`) exigem token válido

### Backend

- Entidade: `Usuario` → tabela `usuarios`
- Migration EF Core aplicada automaticamente no startup (`DatabaseInitializer`)
- Seed automático do usuário de aplicação a partir de `AppUser:Username` e `AppUser:Password`
- Pacote: `Microsoft.AspNetCore.Authentication.JwtBearer`

### Frontend

- `AuthService`: login, validação de expiração, armazenamento do token
- `authInterceptor`: injeta header `Authorization`
- `APP_INITIALIZER`: autentica ao carregar a aplicação

### Endpoints

| Método | Rota | Autenticação |
|--------|------|--------------|
| `POST` | `/api/auth/login` | Pública |
| `GET` | `/health` | Pública |
| `POST` | `/api/contact` | JWT obrigatório |

### Regras

- Não logar tokens nem senhas
- Em produção, usar `Jwt__Secret` forte via variável de ambiente
- Em produção, definir credenciais do usuário de aplicação apenas no servidor (seed) e injetar no build do frontend com cuidado
- Renovar token automaticamente quando expirado (relogin via `AuthService.ensureAuthenticated`)
